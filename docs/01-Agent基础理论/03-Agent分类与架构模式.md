# 一、Agent 分类与架构模式

本章介绍 Agent 的分类和常见架构模式。

## 1. Agent 分类

### 1.1 按能力复杂度分类

| 类型 | 特点 | 示例 |
|------|------|-----|
| **简单反应型** | 简单规则，无记忆 | 规则客服机器人 |
| **状态保持型** | 有短期记忆 | 普通聊天机器人 |
| **目标导向型** | 有目标和规划 | 任务助手 |
| **工具使用型** | 可调用外部工具 | 多工具助手 |
| **多 Agent 协作型** | 多个 Agent 协作 | 团队协作系统 |

### 1.2 按应用场景分类

| 场景 | 描述 | 示例 |
|------|------|-----|
| **客服** | 回答用户问题 | 智能客服 |
| **助手** | 协助完成任务 | 个人助理 |
| **开发** | 协助编程开发 | 代码助手 |
| **研究** | 协助研究工作 | 文献整理、数据分析 |
| **创意** | 生成创意内容 | 写作、画图、作曲 |

## 2. 单 Agent 架构

### 2.1 ReAct 模式

**ReAct** = **Re**asoning + **Act**ing（推理 + 行动）

```
思考 → 行动 → 观察 → 思考 → ... → 回答
```

**流程：**
1. 思考：理解问题，决定下一步
2. 行动：调用工具或操作
3. 观察：获取行动结果
4. 循环：重复直到完成
5. 回答：给出最终答案

**适用场景：** 需要逐步推理、使用工具的任务

### 2.2 Plan-and-Execute 模式

**Plan-and-Execute** = 先规划再执行

```
用户问题
  ↓
规划阶段
  ↓
生成执行计划
  ↓
执行阶段
  ↓
按计划逐步执行
  ↓
回答用户
```

**适用场景：** 复杂任务，需要明确的计划

## 3. 多 Agent 架构

### 3.1 为什么需要多 Agent

| 原因 | 说明 |
|------|------|
| **分工协作** | 不同 Agent 擅长不同任务 |
| **并行处理** | 多个任务可同时进行 |
| **复杂度降低** | 每个 Agent 职责单一 |
| **可扩展性** | 容易添加新 Agent |

### 3.2 常见协作模式

#### 模式 1：主管 + 工人

```
用户问题
  ↓
主管 Agent
  ↓
任务分配
  ├─→ Agent 1: 负责 A
  ├─→ Agent 2: 负责 B
  └─→ Agent 3: 负责 C
  ↓
结果汇总
  ↓
回答用户
```

**适用场景：** 任务可分解为多个子任务

#### 模式 2：串联流程

```
用户问题
  ↓
Agent 1: 分析理解
  ↓
Agent 2: 信息检索
  ↓
Agent 3: 生成回答
  ↓
输出
```

**适用场景：** 线性流程的任务

#### 模式 3：评审反馈

```
用户问题
  ↓
Agent A: 生成初稿
  ↓
Agent B: 评审并给出反馈
  ↓
Agent A: 根据反馈优化
  ↓
(可选) 多轮迭代
  ↓
最终输出
```

**适用场景：** 需要高质量输出的任务

### 3.3 多 Agent 通信

| 通信方式 | 描述 | 特点 |
|---------|------|-----|
| **共享内存** | Agent 读写共享数据区 | 简单，但需处理并发 |
| **消息队列** | Agent 发送/接收消息 | 解耦，异步 |
| **直接调用** | Agent 直接调用其他 Agent | 简单，但耦合高 |
| **事件总线** | 事件订阅/发布 | 灵活，可扩展 |

## 4. 架构模式详解

### 4.1 分层架构

**分层架构** 将系统分为多层，每层负责特定功能。

```
┌─────────────────────────────────────┐
│       用户界面层                     │
├─────────────────────────────────────┤
│       编排层（任务分配、协调）       │
├─────────────────────────────────────┤
│       Agent 层（业务逻辑）           │
├─────────────────────────────────────┤
│       工具层（外部接口）             │
├─────────────────────────────────────┤
│       数据层（存储）                 │
└─────────────────────────────────────┘
```

**优点：**
- 职责清晰
- 易于维护
- 可独立扩展

### 4.2 事件驱动架构

**事件驱动架构** 通过事件来触发和协调各组件。

```
事件源 → 事件总线 → 事件消费者
```

**核心元素：**
- **事件**：发生的事情（如用户提问、工具返回）
- **事件源**：产生事件的组件
- **事件总线**：传递事件的中间件
- **事件消费者**：处理事件的组件

**优点：**
- 解耦
- 灵活扩展
- 异步处理

### 4.3 微服务架构

**微服务架构** 将系统拆分为独立的服务，通过 API 通信。

```
┌─────────┐    ┌─────────┐    ┌─────────┐
│ Agent A │────│ Agent B │────│ Agent C │
└─────────┘    └─────────┘    └─────────┘
     │              │              │
     └──────────┬───┴──────────────┘
                │
         ┌──────▼──────┐
         │  API 网关   │
         └─────────────┘
```

**适用场景：** 大型系统、团队协作

## 5. 在当前项目中的应用

### 5.1 选择建议

| 项目规模 | 推荐架构 |
|---------|---------|
| **小型** | 单 Agent，简单架构 |
| **中型** | 单 Agent + 工具，或简单多 Agent |
| **大型** | 多 Agent + 分层/微服务架构 |

### 5.2 架构演化

建议渐进式演进：

```
阶段 1：简单单 Agent
  ↓
阶段 2：添加工具能力
  ↓
阶段 3：拆分职责，多个 Agent
  ↓
阶段 4：完善架构（分层、事件驱动等）
```
