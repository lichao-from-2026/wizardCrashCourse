# 一、提示词基础

**提示词（Prompt）** 是用户输入给 LLM 的指令或问题，也是 LLM 输出的主要决定因素之一。提示词工程是设计和优化提示词的技术。

## 1. 为什么提示词很重要

### 1.1 LLM 的工作原理

LLM 通过预测下一个最可能的 token 来生成文本，它本质上是在根据提示词"续写"。

```
输入提示词 → LLM 生成续写
```

因此，提示词的质量直接决定了输出的质量。

### 1.2 提示词的作用

| 作用 | 描述 |
|------|------|
| **任务说明** | 告诉 LLM 需要做什么 |
| **信息提供** | 提供必要的背景信息 |
| **格式指定** | 要求 LLM 按特定格式输出 |
| **行为引导** | 引导 LLM 的推理过程 |
| **风格设定** | 设定输出的语气和风格 |

### 1.3 好的提示词 vs 差的提示词

**差的提示词**：
```
写个东西
```

**好的提示词**：
```
你是一个专业的技术文档作者。
请写一篇关于 Python 函数的入门介绍。
要求：
1. 通俗易懂
2. 包含代码示例
3. 字数 500 字左右
```

## 2. 提示词的基本结构

一个完整的提示词通常包含以下要素：

### 2.1 基本结构

```
[角色设定] - 给 LLM 设定角色
[任务描述] - 描述需要完成的任务
[输入数据] - 提供必要的输入
[输出要求] - 指定输出的格式和要求
[示例] - 提供输入输出示例（可选）
[约束条件] - 列出注意事项和约束（可选）
```

### 2.2 各部分详解

#### 2.2.1 角色设定

给 LLM 设定一个角色，让它更专业地完成任务：

**示例**：
```
你是一名经验丰富的软件架构师，擅长设计高可用系统。
```

#### 2.2.2 任务描述

清晰、具体地描述任务：

**示例**：
```
请分析以下系统架构设计，指出潜在的问题并提出改进建议。
```

#### 2.2.3 输入数据

提供需要处理的数据：

**示例**：
```
架构设计文档：
[这里粘贴架构设计文档]
```

#### 2.2.4 输出要求

指定输出的格式和要求：

**示例**：
```
请按以下格式输出：
1. 问题列表（3-5个）
2. 改进建议（对应每个问题）
3. 总体评价
```

#### 2.2.5 示例

提供输入输出示例（Few-shot）：

**示例**：
```
示例 1：
输入：1, 2, 3
输出：[1, 2, 3, 4, 5]

示例 2：
输入：10, 20
输出：[10, 20, 30, 40, 50]
```

#### 2.2.6 约束条件

列出注意事项和约束：

**示例**：
```
注意：
1. 不要使用过于复杂的术语
2. 建议要有可操作性
3. 避免直接修改原架构
```

### 2.3 完整提示词示例

你是一名专业的 Python 教师，擅长用通俗易懂的方式解释编程概念。

请为以下 Python 代码添加详细的注释，并解释代码的功能。

代码：
```python
def calculate_factorial(n):
    if n == 0:
        return 1
    else:
        return n * calculate_factorial(n - 1)

print(calculate_factorial(5))
```

请按以下格式输出：
1. 代码注释（在原代码上添加）
2. 功能解释（100-200字）
3. 时间复杂度分析

注意：
1. 注释要详细但不要冗余
2. 解释要通俗易懂

## 3. 提示词设计原则

### 3.1 核心原则

| 原则 | 描述 |
|------|------|
| **清晰明确** | 避免模糊不清的表述 |
| **具体详细** | 提供足够的细节 |
| **结构合理** | 有清晰的逻辑结构 |
| **示例引导** | 提供示例帮助理解 |
| **逐步引导** | 复杂任务分步骤 |

### 3.2 常见错误和问题

| 错误 | 问题 |
|------|------|
| **过于简单** | 信息不足，LLM 不知道怎么做 |
| **过于冗长** | 信息过多，重要信息被淹没 |
| **模糊不清** | "写点东西"，"分析一下" |
| **矛盾冲突** | 多个要求互相冲突 |
| **过度约束** | 约束太多，限制 LLM 发挥 |

### 3.3 优化建议

**优化前**：
```
帮我写个邮件
```

**优化后**：
```
你是一个专业的商务人士。
请写一封邮件给客户，告知项目将延期一周。
要求：
1. 语气要诚恳
2. 解释延期的原因（供应商延迟）
3. 提出补救方案（优先处理关键功能）
4. 表达歉意并感谢理解
5. 150-200字
```

## 4. 在当前项目中使用提示词工程

在你的 AI Agent 项目中，提示词工程是提升 Agent 表现的关键：

### 4.1 Agent 中的提示词类型

| 提示词类型 | 用途 |
|-----------|------|
| **系统提示词** | 定义 Agent 的角色、目标、行为准则 |
| **工具调用提示词** | 引导 Agent 正确使用工具 |
| **任务提示词** | 针对特定任务设计的提示词 |
| **反思提示词** | 引导 Agent 自我反思和优化 |

### 4.2 最佳实践

1. **模块化**：将提示词分解为可重用的组件
2. **版本管理**：对重要提示词进行版本控制
3. **A/B 测试**：测试不同提示词的效果
4. **收集反馈**：根据实际使用效果持续优化

### 4.3 注意事项

1. **Token 限制**：注意提示词长度，避免超过上下文窗口
2. **成本控制**：高质量提示词可能更长，但效果更好
3. **一致性**：保持提示词风格和要求的一致性
4. **安全性**：防止提示词注入和滥用

## 5. 总结

本章介绍了提示词基础：

- **提示词的重要性**：直接决定 LLM 输出质量
- **基本结构**：角色、任务、输入、输出、示例、约束
- **设计原则**：清晰、具体、结构、示例、逐步
- **在项目中的应用**：系统提示词、工具提示词、任务提示词
