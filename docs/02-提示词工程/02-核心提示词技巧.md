# 一、核心提示词技巧

掌握这些核心技巧，可以显著提升提示词的质量和 LLM 的输出效果。

## 1. 清晰明确的指令

### 1.1 使用明确的动词

避免模糊的表达，使用明确的指令动词。

| 不好的表达 | 更好的表达 |
|-----------|-----------|
| "写点东西" | "写一篇关于 Python 的入门教程" |
| "分析一下" | "分析这段代码的性能问题并提出优化建议" |
| "帮我看看" | "检查这篇文章的语法错误并给出修改建议" |

### 1.2 明确输出要求

清楚告诉 LLM 你期望什么样的输出。

**示例：**
```
写一封邮件给客户，告知项目延期：
- 收件人：客户
- 发件人：项目经理
- 原因：供应商延迟
- 措施：优先处理关键功能
- 字数：200字左右
- 语气：诚恳、专业
```

### 1.3 避免歧义

使用具体的术语，避免模糊的表述。

**不好的示例：**
```
帮我弄个好的东西，不要太复杂
```

**好的示例：**
```
写一个简单的 JavaScript 函数，功能是验证邮箱格式是否合法
```

## 2. 提供充足的上下文

### 2.1 背景信息

提供必要的背景信息帮助 LLM 理解任务。

**示例：**
```
你是一位在电商领域有10年经验的产品经理。
我们的产品是一个在线购物平台，用户主要是25-35岁的年轻人。
现在需要设计一个新的购物车功能，重点提升转化率。
请提出3个核心改进建议，并说明理由。
```

### 2.2 相关数据

如果有相关数据或文档，提供给 LLM。

**示例：**
```
以下是我们公司的销售数据：
- 2024年Q1：100万元
- 2024年Q2：120万元
- 2024年Q3：90万元
- 2024年Q4：150万元

请分析销售趋势，找出可能的原因，并预测2025年Q1的销售额。
```

### 2.3 约束条件

明确列出所有约束条件。

**示例：**
```
请设计一个数据库表结构，满足以下需求：
- 存储用户信息（姓名、邮箱、手机号）
- 支持多语言（中文、英文、日文）
- 需要软删除功能
- 使用 MySQL 数据库
- 遵循第三范式
```

## 3. 指定输出格式

### 3.1 使用 Markdown

要求 LLM 使用 Markdown 格式输出。

**示例：**
```
请写一份项目报告，使用以下格式：

# 项目报告

## 1. 项目概述
- 项目背景
- 目标

## 2. 执行情况
- 已完成工作
- 未完成工作

## 3. 问题与建议
```

### 3.2 使用表格

对于对比性内容，要求使用表格。

**示例：**
```
请对比 React 和 Vue 的主要区别，使用表格格式：

| 维度 | React | Vue |
|------|-------|-----|
| 学习曲线 | | |
| 生态系统 | | |
| 性能 | | |
| 适用场景 | | |
```

### 3.3 使用代码块

对于代码，使用指定语言的代码块。

**示例：**

请写一个 Python 函数，计算斐波那契数列的第 n 项。
使用代码块格式：

```python
def fibonacci(n):
    # 你的代码
```

### 3.4 使用列表

要求使用有序或无序列表。

**示例：**
```
请列出学习 Python 的5个步骤，按重要性排序：

1. 第一步
2. 第二步
3. 第三步
4. 第四步
5. 第五步
```

## 4. 分步骤引导

### 4.1 逐步指示

将复杂任务分解为多个步骤。

**示例：**
```
请按照以下步骤完成任务：

步骤1：理解问题
仔细阅读以下需求，列出关键点。

步骤2：设计方案
提出3个不同的解决方案，分析优缺点。

步骤3：选择方案
从3个方案中选择最佳方案，说明理由。

步骤4：实现方案
写出伪代码或流程图。

需求：[在这里写需求]
```

### 4.2 使用 Chain-of-Thought

引导 LLM 进行思考，而不是直接给出答案。

**示例：**
```
请解决以下问题，先一步步思考，再给出答案。

问题：一个游泳池，单独用A管注水需要6小时，单独用B管注水需要4小时。
如果同时用两个管注水，需要多少小时注满？

思考过程：
1. 首先，计算每小时的注水量
2. A管每小时：1/6
3. B管每小时：1/4
4. 一起每小时：1/6 + 1/4 = 5/12
5. 所以时间：1 / (5/12) = 12/5 = 2.4小时

答案：2.4小时
```

### 4.3 要求解释

要求 LLM 解释答案，帮助你理解。

**示例：**
```
请回答以下问题，并解释你的推理过程：

问题：为什么地球有四季？

要求：
1. 首先给出答案
2. 然后详细解释原因
3. 最后用一个生活中的例子说明
```

## 5. 角色设定

### 5.1 设定专业角色

给 LLM 设定一个角色，让它更专业。

**示例：**
```
你是一位经验丰富的软件架构师，有15年的企业级系统设计经验。
你特别擅长高可用、高性能、可扩展的系统设计。
请帮我设计一个支持百万用户的在线教育平台架构。
```

### 5.2 设定任务背景

结合具体任务设定角色。

**示例：**
```
你是一位在硅谷工作的前端工程师，精通 React、TypeScript、性能优化。
你现在正在为一家初创公司开发产品首页。
请给出你的技术选型和实现方案。
```

### 5.3 设定沟通对象

设定 LLM 要和谁沟通。

**示例：**
```
你需要向一个完全不懂技术的 CEO 解释什么是微服务。
请用通俗易懂的语言，避免技术术语。
可以用生活中的例子来说明。
```

## 6. 在当前项目中的应用

### 6.1 系统提示词模板

在你的项目中，可以建立标准化的系统提示词模板。

**示例模板：**
```
你是一个专业的 [角色]，擅长 [领域]。
你的目标是 [目标]。
请遵循以下原则：
1. [原则1]
2. [原则2]
3. [原则3]

请按以下格式输出：
[输出格式]
```

### 6.2 任务提示词模板

为常见任务建立模板。

**代码审查模板：**
```
请审查以下代码，重点关注：
1. 代码质量和可读性
2. 潜在的 bug 和问题
3. 性能优化建议
4. 安全风险
5. 最佳实践

代码：
[代码]

请按以下格式输出：
## 问题列表
- [问题1]
- [问题2]

## 改进建议
- [建议1]
- [建议2]

## 总体评价
[评价]
```

### 6.3 最佳实践

在当前项目中应用提示词工程的最佳实践：

| 实践 | 说明 |
|------|------|
| **模板化** | 将常用提示词建立模板，避免重复劳动 |
| **版本管理** | 对提示词进行版本管理，记录变更 |
| **A/B 测试** | 测试不同提示词的效果 |
| **收集反馈** | 收集输出结果，持续优化提示词 |
| **审查机制** | 建立提示词审查流程，确保质量 |

### 6.4 注意事项

- 不要过度约束，给 LLM 一定的灵活性
- 测试不同的表达方式，找到最佳效果
- 注意提示词长度，避免超出上下文窗口
- 定期评估和优化提示词

## 7. 总结

核心提示词技巧：
- ✅ 清晰明确的指令
- ✅ 提供充足的上下文
- ✅ 指定输出格式
- ✅ 分步骤引导
- ✅ 角色设定

掌握这些技巧，可以显著提升 LLM 的输出质量！
