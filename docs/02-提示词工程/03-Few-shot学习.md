# 一、Few-shot 学习

Few-shot 学习是指通过给 LLM 提供几个示例，让它理解任务要求并更好地完成任务。

## 1. 什么是 Few-shot 学习

### 1.1 基本概念

| 术语 | 说明 |
|------|------|
| **Zero-shot** | 不给任何示例，直接让 LLM 完成任务 |
| **One-shot** | 给一个示例 |
| **Few-shot** | 给几个示例（通常 2-5 个） |
| **Many-shot** | 给很多示例（通常 10+ 个） |

### 1.2 Few-shot 的优势

相比 Zero-shot，Few-shot 有以下优势：

- ✅ 更准确地理解任务要求
- ✅ 输出格式更一致
- ✅ 减少歧义
- ✅ 提升任务成功率
- ✅ 降低出错概率

**对比示例：**

**Zero-shot：**
```
请分类以下产品：
产品：笔记本电脑
```

**Few-shot：**
```
请分类以下产品：

示例1：
产品：T恤
类别：服装

示例2：
产品：手机
类别：电子产品

示例3：
产品：苹果
类别：食品

请分类：
产品：笔记本电脑
类别：
```

## 2. 示例设计原则

### 2.1 示例要多样化

覆盖不同的情况，让 LLM 理解任务的边界。

**不好的示例（太相似）：**
```
示例1：
输入：苹果
输出：红色

示例2：
输入：草莓
输出：红色

示例3：
输入：樱桃
输出：红色
```

**好的示例（多样化）：**
```
示例1：
输入：苹果
输出：红色

示例2：
输入：香蕉
输出：黄色

示例3：
输入：葡萄
输出：紫色
```

### 2.2 示例要真实

使用真实的数据，避免编造的示例。

**不好的示例（编造）：**
```
示例1：
输入：xyz123
输出：abc456
```

**好的示例（真实）：**
```
示例1：
输入：用户的电子邮件地址
输出：已验证用户身份
```

### 2.3 示例要完整

每个示例都要包含完整的输入和输出。

**不好的示例（不完整）：**
```
示例1：输入：...
示例2：输出：...
```

**好的示例（完整）：**
```
示例1：
输入：问题描述
输出：完整的回答

示例2：
输入：另一个问题
输出：另一个回答
```

### 2.4 示例数量建议

| 任务类型 | 建议示例数量 |
|---------|------------|
| 简单分类 | 2-3 个 |
| 复杂分类 | 3-5 个 |
| 文本生成 | 3-5 个 |
| 代码生成 | 3-5 个 |
| 推理任务 | 5+ 个 |

## 3. Few-shot 示例结构

### 3.1 标准结构

```
任务说明

示例1：
输入：[输入内容]
输出：[输出内容]

示例2：
输入：[输入内容]
输出：[输出内容]

...

现在请处理：
输入：[待处理内容]
输出：
```

### 3.2 带标签的结构

```
任务说明

【示例1】
输入：[输入内容]
输出：[输出内容]

【示例2】
输入：[输入内容]
输出：[输出内容]

【待处理】
输入：[待处理内容]
输出：
```

### 3.3 带解释的结构

```
任务说明

示例1：
输入：[输入内容]
输出：[输出内容]
解释：[为什么这样输出]

示例2：
输入：[输入内容]
输出：[输出内容]
解释：[为什么这样输出]

现在请处理：
输入：[待处理内容]
输出：
```

## 5. 在当前项目中的应用

### 5.1 建立示例库

在你的项目中，可以建立常用任务的示例库。

**示例库结构：**
```
prompts/
├── code-review/
│   ├── examples/
│   │   ├── example1.md
│   │   ├── example2.md
│   │   └── example3.md
│   └── template.md
├── email-writing/
│   ├── examples/
│   └── template.md
└── ...
```

### 5.2 动态选择示例

根据具体任务，动态选择最相关的示例。

**策略：**
```
选择示例的策略：
1. 相似度：选择与当前任务最相似的示例
2. 多样性：覆盖不同的情况
3. 质量：选择输出质量最好的示例
4. 近期：选择最近使用的示例
```

### 5.3 最佳实践

在项目中使用 Few-shot 的最佳实践：

| 实践 | 说明 |
|------|------|
| **版本管理** | 对示例库进行版本管理 |
| **A/B 测试** | 测试不同示例组合的效果 |
| **持续优化** | 根据反馈不断优化示例 |
| **分类管理** | 按任务类型分类管理示例 |

### 5.4 注意事项

- 示例不要太多，避免超过上下文窗口
- 示例质量比数量更重要
- 定期更新示例库
- 注意示例的平衡性

## 6. 总结

Few-shot 学习要点：
- ✅ 提供几个示例帮助 LLM 理解任务
- ✅ 示例要多样化、真实、完整
- ✅ 根据任务类型选择合适的示例数量
- ✅ 在项目中建立示例库并持续优化
