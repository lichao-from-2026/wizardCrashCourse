# 一、上下文窗口管理

理解 LLM 的上下文窗口限制，以及如何有效地管理它。

## 1. 什么是上下文窗口

### 1.1 基本概念

| 概念 | 说明 |
|------|------|
| **上下文窗口** | LLM 一次能处理的最大 Token 数量 |
| **Token** | 文本的基本单位，通常是词或词的一部分 |
| **输入 Token** | 我们输入给 LLM 的内容 |
| **输出 Token** | LLM 生成的内容 |

### 1.2 常见模型的上下文窗口

| 模型 | 上下文窗口 |
|------|-----------|
| GPT-3.5 | 4K / 16K |
| GPT-4 | 8K / 32K / 128K |
| Claude 3 | 200K |
| 长文本模型 | 1M+ |

## 2. Token 估算

### 2.1 估算方法

**大致估算：**
- 1 个 Token ≈ 0.75 个英文单词
- 1 个 Token ≈ 1.3 个中文字符

**示例：**
```
1000 个 Token ≈
- 750 个英文单词
- 1300 个中文字符
```

### 2.2 计算工具

可以使用官方提供的 Token 计算工具：
- OpenAI Tokenizer
- tiktoken（Python 库）
- 各种在线工具

## 3. 上下文窗口管理策略

### 3.1 优先级排序

将最重要的信息放在前面。

**优先级：**
1. 系统提示词（最重要）
2. 用户问题
3. 相关的上下文信息
4. 次要的信息

### 3.2 去重和压缩

删除重复和冗余的信息。

**方法：**
- 删除重复内容
- 压缩冗长的表达
- 只保留关键点
- 使用摘要代替全文

### 3.3 分批处理

当内容太长时，分批处理。

**策略：**
1. 将内容分成多个小部分
2. 依次处理每个部分
3. 最后合并结果

## 4. 在当前项目中的应用

### 4.1 最佳实践

| 实践 | 说明 |
|------|------|
| **估算 Token** | 在发送前估算 Token 数量 |
| **预留空间** | 为输出预留足够的 Token |
| **重要前置** | 重要信息放在前面 |
| **定期清理** | 清理过期的上下文 |

### 4.2 注意事项

- 注意：输入 + 输出 不能超过上下文窗口
- 注意：不同模型的窗口大小不同
- 注意：长上下文可能影响质量
- 注意：成本与 Token 数量成正比

## 5. 总结

上下文窗口管理要点：
- ✅ 理解上下文窗口的限制
- ✅ 学会估算 Token 数量
- ✅ 使用优先级排序
- ✅ 去重和压缩内容
- ✅ 必要时分批处理
